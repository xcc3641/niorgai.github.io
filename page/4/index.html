Loading界面简单实现

发表于 2015 Mar 2   |   分类于 Android开发tips   |  

应用经常会涉及一些耗时操作，尽管经常会开新线程去处理，但总是会耗时。所以就希望有一个loading界面,
想来想去觉得使用自定义的Dialog比较好。

思路：

  1. 使用Dialog，自定义其布局。
  2. 布局中使用ProgressBar实现旋转的Loading。
  3. 通过setText方法可以设置文字

LoadingDialog类

public class LoadingDialog extends Dialog{
    private LinearLayout parentLayout;
    private TextView loadingText;

    public LoadingDialog(Context context) {
        super(context,R.style.loading_dialog);
        init(context);
    }

    public LoadingDialog(Context context, int theme) {
        super(context, R.style.loading_dialog);
        init(context);
    }

    private void init(Context context){
        this.setContentView(LayoutInflater.from(context).inflate(R.layout.loading_dialog, null));
        parentLayout = (LinearLayout) findViewById(R.id.dialog_view);
        loadingText = (TextView) findViewById(R.id.text);
        this.setCancelable(true);
    }

    //设置名字和外层背景色
    public void setText(String text, int color){
        if(!TextUtils.isEmpty(text)) {
            parentLayout.setBackgroundColor(color);
            loadingText.setText(text);
            loadingText.setVisibility(View.VISIBLE);
        }else {
            loadingText.setVisibility(View.GONE);
        }
    }
}

Android拍照选取图片

发表于 2015 Jan 31   |   分类于 Android开发经验   |  

Android经常会需要拍照、裁剪及图库中选择图片，其实都是通过intent调用系统相机或者系统图册，然后在onActivityResult中捕捉返回即可。

正常拍照选择图片的代码：
Intent intent = new Intent(MediaStore.ACTION_IMAGE_CAPTURE);

在onActivityResult中通过返回的intent.getExtras().get("data")便可以获取图片，但Android默认可用的应用内存大约为16M.所以Android为了不超出内存限制，在拍照返回时通过intent返回的bitmap会被压缩，这样导致一直都是获取的小图。所以在拍照时如果想返回大图需要通过Uri返回。

拍照选择大图代码(tampUri为路径，开发者文档提供的代码)：
Intent intent = new Intent(MediaStore.ACTION_IMAGE_CAPTURE);
// 指定调用相机拍照后照片的储存路径
intent.putExtra(MediaStore.EXTRA_OUTPUT, tempUri);

在onActivityResult中通过tempUri进行下一步操作。

说明：

拍照获取的图片推荐存放在 Activity.this.getExternalCacheDir()返回的/storage/sdcard0/Android/data/<package name>/cache路径，通过new File(path,fileName)新建一个file，然后通过Uri.fromFile(file)即可获取正确的uri，记得fileName需要带后缀。

为了防止Uri路径不对导致不能保存，使用前必须确保文件路径无误，不然会导致“无法保存剪裁的图片”的错误。或者无提示但不能点击确定保存图片。正确获取的Uri最终以file://开头。。

uri获取bitmap对象的方法可以参考Bitmap相关:读取大图片。

代码示例：

String fileName ＝ "temp.jpg";
File tmpFile = new File(this.getExternalCacheDir(),fileName);
Uri tmpUri = Uri.fromFile(tmpFile);
  1. 设置通过Uri返回后，onActivityResult中返回的intent为null。

  2. 开发者文档中只说了简单调用的方法，其实可以添加一些其他属性。

    intent.putExtra("return-data",false);
intent.putExtra("outputFormat",Bitmap.CompressFormat.JPEG.toString());
intent.putExtra("noFaceDetection", true);
  3. 不能添加intent.setType("image/*");，会因为找不到intent导致ActivityNotFoundException。
  4. intent.putExtra("android.intent.extras.CAMERA_FACING",1);，系统默认开启的是后置摄像头，如果希望选择前置摄像头可以加这句。

目前从图库选择图片可以有两种方法：

以下结果属于vivo Xplay,Android Version 4.4.4测试结果，如果有其他机型测试我会补充
1.Intent.ACTION_PICK：

Intent intent = new Intent(Intent.ACTION_PICK,
            android.provider.MediaStore.Images.Media.EXTERNAL_CONTENT_URI);
startActivityForResult(intent, PHOTO_PICK);

onActivityResult返回的Intent中：

  • intent.getData()会返回所选图片的Uri，格式为content://media/external/images/media/919
  • intent.getExtras()返回的bundle中包含一个boolean变量，通过data.getExtras().get("isFinishGallery")获取,应该是指是否获取完毕，查询文档没有官方结果。
  • intent.putExtra("return-data",false)对返回值没有影响。

2.Intent.ACTION_GET_CONTENT

Intent intent = new Intent(Intent.ACTION_GET_CONTENT);
intent.setType("image/*");//不加会导致crash

onActivityResult返回的Intent中：

  • intent.getData()返回所选图片的Uri：

关于返回Uri格式的说明：

  • 系统版本大于等于19(KitKat):content://com.android.providers.media.documents/document/image%3A919

  • 系统版本低于19:content://media/external/images/media/919

可以通过以下函数获取真实路径：

private String getRealPathFromURI(Context context,Uri contentUri) {
    String[] proj = { MediaStore.Images.Media.DATA };
    Cursor cursor = context.getContentResolver().query(contentUri, proj, null, null, null);
    if (cursor != null) {
        cursor.moveToFirst();
        int columnIndex = cursor.getColumnIndex(proj[0]);
        String path = cursor.getString(columnIndex);
        cursor.close();
        return path;
    }
    return contentUri.getPath();
   }

Android序列化对象

发表于 2015 Jan 12   |   分类于 Android基础知识   |  

默认情况下，Activity之间只能传递基本数据类型，如果需要传递某个对象，则需要将其序列化，Android中序列化便是实现Serializable或则Parcelable接口。

序列化:永久性保存对象,即保存对象的字节序列到本地文件中。

Serializable的作用是将对象的属性保存到本地文件、数据库、网络流以方便传输。Parcelable的设计初衷是因为Serializable的效率太慢，为了**在程序内不同组件间及不同Android程序间(AIDL)高效传输而设计。

使用的选择：

  1. 在内存间传递数据时，推荐使用Parcelable，因为其内存开销小，性能高。
  2. 在需要保存时，推荐使用Serializable，因为Parcelable不适合做持久化存储。
  3. Serializable使用IO读写存储在硬盘上，而Parcelable是直接在内存中读写，很明显内存的读写速度通常大于IO读写，所以在Android中通常优先选择Parcelable。

接下来利用一个例子说明如何实现这些接口。首先是初始对象：

public class Test {
    int num;
    String string;
}

如果实现Serializable接口，只需要继承java.io.Serializable，并提供一个serialVersionUID即可。

public class Test implements Serializable {

    final long serialVersionUID = 12345678l;
    int num;
    String string;
}

如果实现Parcelable,需要继承android.os.Parcelable，重写describeContents方法和writeToParcel方法，并定义一个Parcelable.Creator类型的静态变量，在该静态变量中实现readFromParcel方法(自己定义的名字)。

注意：

  1. writeToParcel方法中将变量按顺序写入Parcel

  2. Parcelable.Creator的createFromParcel方法，从Parcel读入变量时必须按照writeToParcel方法中的顺序读入。

    public class Test implements Parcelable{
    int num;
    String string;

    @Override
    public int describeContents() {
            return 0;
        }

        @Override
        public void writeToParcel(Parcel dest, int flags) {
            dest.writeInt(num);
             dest.writeString(string);
        }

    public static Creator<Test> creator = new Creator<Test>() {
            @Override
        public Test createFromParcel(Parcel source) {
            return new Test(source);
        }

        @Override
        public Test[] newArray(int size) {
            return new Test[size];
        }
    };

    Test(){}

    Test(Parcel source){
                readFromParcel(source);
        }

    private void readFromParcel(Parcel source){
        //读取对象数据，必须按照writeToParcel方法中
        //写入数据的顺序依次读取数据
        num = source.readInt();
        string = source.readString();
    }

}

  3. 如果需要序列化自定义的数据，该数据也需要实现Parcelable接口。

    例如希望通过序列化类TestA，类TestA中有一个List存放类TestB。

    class TestA{
    List<TestB> list;
}

    那么TestA与TestB都需要实现Parcelable接口。且类TestA通过parcel.writeTypedList写入list，通过parcel.readTypedList(list,TestB.creator)读取list。

Android代码混淆

发表于 2014 Dec 26   |   分类于 Android开发经验   |  

在这篇文章中说了反编译Android APK的方法，当然作为开发者很多时候不希望某些涉及机密的代码被别人看到，于是需要在打包APK的时候做一些代码混淆。

混淆技巧(感谢Android打包的那些事):

  1. 默认的混淆:

    sdk目录下的android-sdk/tools/proguard/proguard-android.txt文件包含了buildTools对应版本的混淆配置,不需要再改动.

  2. project路径下的proguard-rules.pro文件(最新Android Studio新建的工程可能不叫这个名字)对应自己项目的混淆配置.

    1. 第三方的包可以直接不混淆包名.如友盟(com.umeng):

      -dontwarn com.umeng.**
-keep class com.umeng.** { *; }

    2. 混淆后友盟错误统计需要上传mapping文件,追查bug很不好,所以建议开启行数.

      #不混淆行数,方便debug
-keepattributes SourceFile,LineNumberTable

  3. 其他技巧,这都是可以在proguard的文档上查到的.

    #指定代码的压缩级别
-optimizationpasses 5

#包明不混合大小写
-dontusemixedcaseclassnames

#不去忽略非公共的库类
-dontskipnonpubliclibraryclasses

 #不优化输入的类文件
-dontoptimize

 #预校验
-dontpreverify

 #混淆时是否记录日志
-verbose

 # 混淆时所采用的算法
-optimizations !code/simplification/arithmetic,!field/*,!class/merging/*

#保护注解
-keepattributes *Annotation*

#忽略警告
-ignorewarning

#apk 包内所有 class 的内部结构
-dump class_files.txt
#未混淆的类和成员
-printseeds seeds.txt
#列出从 apk 中删除的代码
-printusage unused.txt
#混淆前后的映射
-printmapping mapping.txt
